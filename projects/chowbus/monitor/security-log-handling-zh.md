## 1. 执行摘要

### 目的

本文档对 Monitor 系统的敏感日志处理进行安全评估，涵盖以下方面：

- 日志如何按敏感级别分类
- 保护敏感数据的访问控制措施
- 数据脱敏和加密机制
- 当前数据隔离架构

### 主要发现

| 控制领域 | 当前成熟度 | 风险等级 |
|---------|-----------|---------|
| 数据分类 | 已实现 | 低 |
| 身份认证 | 强 | 低 |
| 访问授权 | 弱 | **高** |
| 数据脱敏 | 强 | 低 |
| 静态加密 | 未实现 | **中** |
| 审计日志 | 部分实现 | **中** |

## 3. 日志分类框架

### 3.1 当前分类模型

系统实现了**三级敏感度模型**：

| 级别 | 描述 | 保留期限 | 存储位置 |
|-----|------|---------|---------|
| **高 (HIGH)** | PII、财务数据、认证凭据 | 180天 | 独立表 |
| **中 (MEDIUM)** | 联系方式、行为数据、设备指纹 | 28天 | 普通表 |
| **低 (LOW)** | 运营指标、非PII元数据 | 28天 | 普通表 |

### 3.2 分类标准

#### 高敏感度（需要最高级别保护）

| 数据类型 | 示例 | 合规关注点 |
|---------|------|-----------|
| 支付卡数据 | 完整卡号、CVV、有效期 | PCI-DSS |
| 政府证件号 | SSN、驾照号、护照号 | 身份盗窃风险 |
| 认证凭据 | 密码、令牌、API密钥 | 账户被盗风险 |
| 金融账户号 | 银行账号、路由号码 | 金融欺诈风险 |
| 健康信息 | 医疗记录、处方 | HIPAA（如适用） |

#### 中敏感度（需要保护）

| 数据类型 | 示例 | 合规关注点 |
|---------|------|-----------|
| 联系信息 | 邮箱、电话、地址 | 隐私法规 |
| 行为数据 | 购买历史、偏好 | 用户画像问题 |
| 位置数据 | GPS坐标、IP定位 | 隐私法规 |
| 设备指纹 | User Agent、设备ID组合 | 跟踪问题 |

#### 低敏感度（标准保护）

| 数据类型 | 示例 |
|---------|------|
| 运营指标 | 响应时间、错误计数 |
| 聚合统计 | 订单量、流量模式 |
| 技术元数据 | 请求路径、状态码 |

### 3.3 分类机制

| 机制 | 描述 |
|-----|------|
| 敏感度枚举 | `SensitiveLevel` 枚举定义 LOW/MEDIUM/HIGH 三级 |
| 业务动作配置 | 每个业务动作可配置对应的敏感级别 |
| Groovy 脚本 | 通过脚本在日志解析时动态设置敏感级别 |
| 双表存储 | 高敏感日志自动路由到独立表 |

### 3.4 当前缺口

| 缺口 | 描述 |
|-----|------|
| 仅支持手动分类 | 敏感级别需人工在业务动作配置中设置 |
| 无自动PII检测 | 不会自动扫描日志内容识别敏感数据 |
| 静态分配 | 一旦分类，不会根据内容重新分类 |

## 4. 访问控制评估

### 4.1 身份认证控制

| 控制项 | 实现方式 | 评估 |
|-------|---------|-----|
| 认证方式 | JWT令牌 + SSO | **强** |
| 身份提供商 | Okta集成 | **强** |
| 会话管理 | 令牌制，24小时过期 | **充分** |
| 多因素认证 | 通过Okta强制执行 | **强** |

### 4.2 授权控制

#### 当前角色模型

| 角色 | 预期用途 | 当前权限 |
|-----|---------|---------|
| SUPER_ADMIN | 完全系统控制 | 所有操作 |
| ADMIN | 管理功能 | 所有操作 |
| USER | 标准访问 | 所有读取操作 |
| EXTERNAL | 合作伙伴访问 | 所有读取操作 |

#### 当前缺口

| 发现 | 严重程度 | 描述 |
|-----|---------|-----|
| **无数据级授权** | **严重** | 所有认证用户可访问所有日志数据，不论敏感度 |
| **角色无差异** | **高** | USER和EXTERNAL权限相同 |
| **无查询限制** | **高** | 对数据量或查询范围无限制 |
| **缺少审计追踪** | **中** | 敏感数据访问未记录 |

## 5. 数据保护控制

### 5.1 数据脱敏

#### 当前实现

| 能力 | 状态 | 评估 |
|-----|-----|-----|
| 字段级脱敏 | 已实现 | **强** |
| JSON路径支持 | 已实现 | **强** |
| 可配置规则 | 已实现 | **强** |
| 多种脱敏类型 | 5种可用 | **强** |

#### 可用的脱敏类型

| 类型 | 使用场景 | 示例 |
|-----|---------|-----|
| 中间脱敏 | 电话号码 | 138****5678 |
| 前缀脱敏 | 邮箱地址 | ***an@email.com |
| 后缀脱敏 | 地址 | 北京市朝阳区*** |
| 全部脱敏 | 密码、令牌 | ******** |
| 模式脱敏 | 信用卡 | 4111-****-****-1111 |

#### 脱敏规则管理

| 配置项 | 描述 |
|-------|------|
| `data_masking_rule` 表 | 存储脱敏规则配置 |
| 字段路径 | 支持 JSON 路径如 `input.phone`、`response.data.email` |
| 数组支持 | 支持 `items[*].phone` 通配符匹配数组元素 |
| 业务动作绑定 | 规则按业务动作ID关联 |

#### 当前缺口

| 发现 | 严重程度 | 描述 |
|-----|---------|-----|
| **仅查询时脱敏** | **高** | 原始数据未脱敏存储在数据库中 |
| **无入库时脱敏** | **高** | PII在存储层持久化 |
| **数据库管理员绕过脱敏** | **中** | 直接数据库访问可见原始数据 |

### 5.2 加密现状

| 层级 | 当前状态 | 风险 |
|-----|---------|-----|
| **传输中（外部）** | 强制TLS 1.2+ | 低 |
| **传输中（内部）** | 服务间使用HTTP | 中 |
| **静态（数据库）** | 未加密 | **中** |
| **字段级加密** | 未实现 | 中 |

## 6. 数据隔离现状

### 6.1 当前架构

系统采用**单实例双表**架构进行数据隔离：

```
  ┌───────────────────────────────────────────────────────┐
  │              StarRocks 单一实例                        │
  │                                                        │
  │   数据库：monitor_db                                   │
  │                                                        │
  │   ┌─────────────────────────────────────────────────┐ │
  │   │ 表：business_action_logs                        │ │
  │   │                                                 │ │
  │   │ • 存储 LOW 和 MEDIUM 敏感级别日志               │ │
  │   │ • 保留期限：28天                                │ │
  │   │ • 分区策略：按 client_time 按天分区             │ │
  │   │ • 分布策略：Hash(restaurant_id, trace_id)       │ │
  │   └─────────────────────────────────────────────────┘ │
  │                                                        │
  │   ┌─────────────────────────────────────────────────┐ │
  │   │ 表：business_action_logs_sensitive              │ │
  │   │                                                 │ │
  │   │ • 存储 HIGH 敏感级别日志                        │ │
  │   │ • 保留期限：180天                               │ │
  │   │ • 分区策略：按 client_time 按天分区             │ │
  │   │ • 分布策略：Hash(restaurant_id, trace_id)       │ │
  │   └─────────────────────────────────────────────────┘ │
  │                                                        │
  └───────────────────────────────────────────────────────┘
```

### 6.2 隔离机制

| 机制 | 描述 |
|-----|------|
| **表级隔离** | 高敏感日志存储在独立表中 |
| **自动路由** | 日志采集服务根据 `sensitive_level` 字段自动路由 |
| **差异化保留** | 敏感表保留180天，普通表保留28天 |
| **相同Schema** | 两表结构相同，便于统一查询逻辑 |

### 6.3 当前访问控制

| 层级 | 当前状态 |
|-----|---------|
| **网络层** | 单一VPC，无网络隔离 |
| **数据库层** | 单一数据库用户，无RBAC |
| **应用层** | 无敏感表专用API端点 |
| **审计层** | 无敏感数据访问日志 |

### 6.4 StarRocks 安全能力

StarRocks 支持以下安全特性（当前未启用）：

| 特性 | 描述 | 当前状态 |
|-----|------|---------|
| 用户认证 | 内部用户或LDAP集成 | 使用单一root用户 |
| 数据库级权限 | `GRANT SELECT ON db.* TO role` | 未配置 |
| 表级权限 | `GRANT SELECT ON db.table TO role` | 未配置 |
| 审计日志 | 查询日志记录 | 未启用 |
