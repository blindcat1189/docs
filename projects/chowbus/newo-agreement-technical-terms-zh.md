# Chowbus - Newo.ai 协议：技术要求

> 技术义务合同条款（基于行业标准修订）

---

## 1.3 Newo.ai 技术义务

Newo.ai 应：

### 1.3.1 API 访问
- 提供 AI Agent 平台的 REST API 访问
- 提供 API 文档（OpenAPI/Swagger 规范）
- 维护 API 版本控制，重大变更需提前至少 12 个月通知
- 非重大变更需保持向后兼容性
- 支持每家餐厅每分钟至少 1,000 次 API 请求

### 1.3.2 集成支持
- 提供以下响应和解决时间的技术支持：

| 优先级 | 定义 | 响应时间 | 解决目标 |
|--------|------|----------|----------|
| 紧急 | 服务不可用 | 1 小时 | 4 小时 |
| 高 | 服务降级 | 4 小时 | 8 小时 |
| 中 | 功能受损 | 8 小时 | 48 小时 |
| 低 | 一般咨询 | 24 小时 | 5 个工作日 |

### 1.3.3 Agent 配置
- 在 API 请求后 5 分钟内配置新的 AI agent
- 支持多门店餐厅的批量配置
- 提供配置后 24 小时内的回滚能力

### 1.3.4 白标方案
- 支持完整的 UI 品牌重塑，无需 Newo.ai 署名
- 提供可定制的脚本（问候语、结束语、错误处理）
- 支持自定义语音/人设配置

### 1.3.5 报告和分析
- 提供通话指标和性能的实时仪表板
- 提供使用和分析数据的 API 访问
- 报告数据至少保留 24 个月

### 1.3.6 多语言支持
- 上线时支持英语、中文（普通话）和西班牙语
- 所有支持语言的准确率保持一致（差异在 5% 以内）
- 额外语言需双方书面同意

## 1.4 集成要求

### 1.4.1 数据同步

| 数据类型 | 同步方式 | 最大延迟 | 重试策略 |
|----------|----------|----------|----------|
| 餐厅信息 | 拉取 + Webhook | 5 分钟 | 3 次重试，指数退避 |
| 菜单结构 | 拉取 + Webhook | 5 分钟 | 3 次重试，指数退避 |
| 菜品可用性 | Webhook | 30 秒 | 3 次重试，指数退避 |
| 价格变更 | Webhook | 60 秒 | 3 次重试，指数退避 |
| 订单状态 | Webhook | 5 秒 | 3 次重试，指数退避 |

### 1.4.2 订单提交
- 在客户确认后 2 秒内向 Chowbus API 提交订单
- 每个订单请求包含幂等键（UUID v4）
- 实现重试逻辑：3 次尝试，指数退避（1秒、2秒、4秒）
- 所有订单尝试和响应日志至少保留 90 天

### 1.4.3 Webhook 实现
- 在 5 秒内处理传入的 webhook
- 成功接收返回 HTTP 2xx
- 使用 HMAC-SHA256 验证 webhook 签名
- 实现重复 webhook 的幂等处理

### 1.4.4 电话服务
- 为每个餐厅位置配置专用电话号码
- 电话号码归 Chowbus 所有，由 Newo.ai 管理
- 支持在终止请求后 30 天内完成号码携转
- 在服务中断期间实现呼叫转移到指定备用号码

### 1.4.5 客户通知
- 在订单成功提交后 30 秒内发送订单确认
- 在收到 webhook 后 60 秒内发送状态更新
- 支持按餐厅配置的短信和/或语音回拨

## 1.5 服务级别协议

### 1.5.1 可用性目标

| 服务 | 月度正常运行时间 | 每月最大停机时间 |
|------|------------------|------------------|
| AI Agent 平台 | 99.9% | 43 分钟 |
| 管理门户 | 99.5% | 3.6 小时 |
| 电话/通信服务 | 99.99% | 4.3 分钟 |

**测量方式**：正常运行时间由 Newo.ai 监控系统测量。Chowbus 可要求第三方验证。

### 1.5.2 SLA 排除项

以下情况不计入停机时间：
- 计划维护（提前 72 小时通知）
- Chowbus 系统不可用
- 不可抗力事件
- Chowbus 滥用服务导致的问题
- Newo.ai 网络外的互联网连接问题

### 1.5.3 性能目标

| 指标 | 目标 | 测量方式 |
|------|------|----------|
| AI 语音响应延迟 | < 1.5 秒 (p95) | 首次 AI 响应时间 |
| 订单 API 响应 | < 2 秒 (p95) | API 往返时间 |
| Webhook 处理 | < 5 秒 (p95) | 接收到确认 |
| 语音识别准确率 | > 95% | 月度抽样审计 |
| 订单准确率 | > 99% | 无错误订单 / 总订单 |

### 1.5.4 服务积分

| 月度正常运行时间 | 积分（月费百分比） |
|------------------|-------------------|
| 99.5% - 99.9% | 5% |
| 99.0% - 99.5% | 10% |
| 95.0% - 99.0% | 25% |
| < 95.0% | 50% |

**积分上限**：任何日历月内的最高服务积分不超过月费的 50%。

**申请流程**：Chowbus 须在事件发生后 30 天内提交积分申请。Newo.ai 应在 14 天内回复。

**唯一救济**：服务积分是 Chowbus 因未达到 SLA 目标的唯一和排他性救济，重大过失或故意不当行为除外。

### 1.5.5 计划维护

- 至少提前 72 小时通知（发送至指定联系人邮箱）
- 首选时间窗口：太平洋时间凌晨 2:00 - 6:00
- 每个日历月最多 4 小时计划维护
- 紧急维护可不提前通知；1 小时内发出通知

## 1.6 安全要求

### 1.6.1 数据传输
- 所有 API 和 webhook 通信使用 TLS 1.2 或更高版本
- 建议移动/嵌入式客户端使用证书锁定
- Webhook 签名验证至少使用 HMAC-SHA256

### 1.6.2 数据存储
- 静态数据使用 AES-256 或同等加密
- 加密密钥通过硬件安全模块（HSM）或同等设施管理
- 数据库访问仅限授权人员

### 1.6.3 访问控制
- 所有系统实施基于角色的访问控制（RBAC）
- 以下情况要求多因素认证（MFA）：
  - 管理访问
  - 生产系统访问
  - 客户数据访问
- 使用唯一用户账户（禁止共享凭证）
- 每季度进行访问审查

### 1.6.4 审计日志
- 记录所有客户数据访问
- 记录所有管理操作
- 日志保留：至少 12 个月
- 日志防篡改并集中存储

### 1.6.5 安全认证
- **SOC 2 Type II**：维护当前认证；每年提供报告
- **渗透测试**：每年由合格第三方进行；分享执行摘要
- **漏洞扫描**：至少每季度进行；7 天内修复严重发现

### 1.6.6 事件响应
- 维护书面的事件响应计划
- 发现安全事件后 24 小时内通知 Chowbus
- 解决后 7 天内提供事后报告

## 1.7 数据保护

### 1.7.1 数据所有权
- **客户数据**：从终端客户收集的所有数据（姓名、电话、订单）归 Chowbus 所有
- **使用数据**：Newo.ai 可保留汇总的匿名使用统计数据
- **AI 模型**：Newo.ai 拥有 AI 模型；未经匿名化处理，Chowbus 数据不得用于为其他客户训练模型

### 1.7.2 数据处理
- Newo.ai 作为**数据处理者**；Chowbus 作为**数据控制者**
- 仅按 Chowbus 指示并在提供服务所需范围内处理数据
- 数据处理协议（DPA）作为附件 A 附后

### 1.7.3 子处理者
- Newo.ai 应维护子处理者清单并应要求提供
- 启用新子处理者前需提前 30 天通知
- Chowbus 可在合理理由下反对新子处理者

### 1.7.4 数据保留

| 数据类型 | 保留期限 | 删除触发条件 |
|----------|----------|--------------|
| 通话录音 | 90 天 | 期满自动删除 |
| 通话转录 | 90 天 | 期满自动删除 |
| 订单数据 | 30 天 | 传输至 Chowbus 后 |
| 客户 PII | 30 天 | 交易完成后 |
| 汇总分析 | 24 个月 | 不适用（已匿名化） |

### 1.7.5 数据主体权利
- 支持 Chowbus 响应数据主体请求（访问、删除、可携性）
- 在 Chowbus 请求后 72 小时内响应
- 合理数量的请求不收取额外费用

### 1.7.6 数据泄露通知
- 发现确认的数据泄露后 **24 小时**内通知 Chowbus
- 发现疑似泄露后 **48 小时**内通知 Chowbus
- 通知应包括：泄露性质、受影响数据、补救措施、时间表

## 1.8 业务连续性

### 1.8.1 灾难恢复

| 指标 | 目标 |
|------|------|
| 恢复时间目标（RTO） | 4 小时 |
| 恢复点目标（RPO） | 1 小时 |

### 1.8.2 备份
- 每日备份所有配置和客户数据
- 备份存储在地理位置不同的区域
- 每季度测试备份恢复

### 1.8.3 冗余
- 关键服务无单点故障
- AI Agent 平台采用多区域或多可用区部署

## 1.9 审计权利

### 1.9.1 审计范围
Chowbus 可审计 Newo.ai 对以下方面的合规性：
- 安全要求（第 1.6 节）
- 数据保护义务（第 1.7 节）
- 服务级别承诺（第 1.5 节）

### 1.9.2 审计流程
- **频率**：每日历年一次，或在合理怀疑违规时
- **通知**：提前 30 天书面通知（怀疑违规时除外）
- **进行方式**：在正常工作时间内，尽量减少干扰
- **费用**：各方承担各自费用
- **保密性**：审计发现作为保密信息处理

### 1.9.3 第三方审计
- Chowbus 可依赖 SOC 2 Type II 报告代替直接审计
- Chowbus 可聘请合格的第三方审计师

## 1.10 终止义务

终止或到期时，Newo.ai 应：

| 义务 | 时间要求 |
|------|----------|
| 继续服务（过渡期） | 通知后 90 天 |
| 提供数据导出（通话日志、转录、分析） | 请求后 14 天内 |
| 将电话号码携转至 Chowbus 指定运营商 | 请求后 30 天内 |
| 删除所有 Chowbus 客户数据 | 终止后 30 天内 |
| 提供书面删除证明 | 删除后 30 天内 |
| 归还或销毁保密信息 | 终止后 30 天内 |

### 1.10.1 过渡协助
- Newo.ai 应在过渡期内提供合理的过渡协助
- 协助包括：数据导出、文档、技术支持
- 超过 20 小时的协助可能需要额外费用

## 定义

| 术语 | 定义 |
|------|------|
| **停机时间** | 服务不可用的时段，不包括 SLA 排除项 |
| **月度正常运行时间** | （总分钟数 - 停机分钟数）/ 总分钟数 × 100% |
| **p95** | 测量值的第 95 百分位数 |
| **客户数据** | 通过服务收集的终端客户数据 |
| **PII** | 个人身份信息 |
| **RTO** | 恢复时间目标 - 灾难后恢复服务的最长时间 |
| **RPO** | 恢复点目标 - 以时间衡量的最大可接受数据损失 |
